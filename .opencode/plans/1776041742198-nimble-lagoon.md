# Rebuild Home As Minimal Posts Index

## Goal
Replace the current splash-style home page with a minimal, production-ready posts index that:
- fetches posts from `https://cuevaio.substack.com/feed`
- merges them with local manually curated items from a JSON file
- revalidates once per day
- uses a restrained, aligned editorial layout like the reference
- keeps only the X link in the top nav
- moves the other links to the footer
- applies the same minimalist layout principles to `/about`

## Recommended Approach

### 1. Add a small server-side posts data layer
Add:
- `lib/posts.ts`
- `data/posts.json`

Implementation:
- Fetch the Substack RSS feed from `https://cuevaio.substack.com/feed` on the server.
- Add `rss-parser` as the only new data dependency.
- Normalize RSS items into a shared shape, for example:
  - `date`
  - `title`
  - `link`
  - optional `source`
- Store extra curated entries in `data/posts.json` using the same shape.
- Merge RSS items and local JSON items, then sort newest-first by normalized timestamp.

### 2. Add daily revalidation to the home route
Modify:
- `app/page.tsx`

Implementation:
- Convert `app/page.tsx` into an async server page.
- Add `export const revalidate = 86400`.
- In `lib/posts.ts`, also use `fetch(feedUrl, { next: { revalidate: 86400 } })` for clarity.

### 3. Replace the current home splash with a minimal index layout
Modify:
- `app/page.tsx`

Implementation:
- Remove:
  - `CustomCursor`
  - glitch hero heading
  - inline icon social nav
  - centered landing-page composition
- Keep the current dark visual language:
  - `ShaderBackground`
  - `GrainOverlay`
- Rebuild the page as:
  - `SiteNav`
  - a single aligned content column
  - a plain posts list
  - `SiteFooter`
- Use a shared width rhythm such as `max-w-5xl` outer shell with a narrower inner content column.
- Keep alignment consistent from header to content to footer.

### 4. Make the home page list minimal and structured
Modify/add:
- `app/page.tsx`
- optionally `components/posts/post-list.tsx`

Implementation:
- Render one row per item with a minimal layout like the reference:
  - date in a muted, narrow column
  - title as the main clickable content
  - no extra card UI
- Keep spacing consistent and understated.
- Use serif typography to match the rest of the app.
- Preserve the dark background and avoid decorative containers.

### 5. Simplify the shared nav and footer
Modify:
- `components/site-nav.tsx`
- `components/site-footer.tsx`

Implementation:
- `SiteNav`:
  - keep brand/home link on the left
  - keep only `About` and `X` on the right, with X as the only external social link at the top
  - remove GitHub and Email from the header
- `SiteFooter`:
  - move the remaining links here: GitHub, LinkedIn, Instagram, Email
  - keep it sparse and aligned with the page content width
- Use the same nav/footer shell on both `/` and `/about`.

### 6. Bring `/about` into the same minimalist system
Modify:
- `app/about/page.tsx`
- `components/about/about-content.tsx`
- `components/about/markdown-components.tsx`

Implementation:
- Keep `about.md` as the content source.
- Keep semantic markdown rendering with `react-markdown`.
- Keep the portrait expand toggle, but remove any remaining prototype/demo feel.
- Make `/about` use the same page shell as the new homepage:
  - same nav
  - same width rhythm
  - same footer
  - no gray content panel
  - transparent content area inheriting the page background
- Keep body prose serif and restrained.
- Tighten spacing so `/about` feels like a calm editorial page rather than a separate special layout.

### 7. Preserve app-wide serif typography
Modify only if needed:
- `app/layout.tsx`
- `components/about/markdown-components.tsx`

Implementation:
- Keep serif as the app-wide default.
- Ensure homepage list items and about prose both inherit the serif styling cleanly.
- Avoid mixing in sans body copy on these two pages unless there is a strong readability reason.

### 8. Keep markdown import support as-is
Keep:
- `next.config.mjs`
- `md.d.ts`

Implementation:
- Continue importing `about.md` directly in the server page.
- No filesystem reads are needed for `/about` content.
